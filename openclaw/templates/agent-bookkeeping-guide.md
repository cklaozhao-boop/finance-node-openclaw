# __AGENT_NAME__ 记账使用说明

这份文档给两类读者用：

- 你本人：知道该怎么和 `__AGENT_NAME__` 说，才能稳定记账
- `__AGENT_NAME__` 自己：启动后能快速定位默认入口、默认规则和输出格式

## 默认原则

- 当前项目里，`__AGENT_NAME__` 是默认记账入口。
- 所有收入、支出、内部转账、账户余额联动，都以本地 Finance Node 为唯一真源。
- 所有记账都只写本地 Finance Node。
- 即使用户是从飞书、Discord 或微信来对话，记账后端仍然是 Finance Node。

## 记账入口位置

- 技能说明：`skills/finance-node-bookkeeper/SKILL.md`
- 实际写入脚本：`skills/finance-node-bookkeeper/scripts/finance_node_cli.py`
- 节点运行配置：由安装器自动写入 `${XDG_CONFIG_HOME:-~/.config}/finance-node-openclaw/install.json`

## 你应该怎么和 __AGENT_NAME__ 说

最稳的方式是一次说清楚这些字段：

- 类型：收入 / 支出 / 转账
- 金额
- 账户
- 资金来源或类别
- 项目
- 时间
- 用途或备注

## 记账标准

### 支出

最少要有：

- 时间
- 金额
- 支出账户
- 项目
- 类别
- 备注或用途

### 收入

最少要有：

- 时间
- 金额
- 入账账户
- 资金来源
- 备注或来源说明

### 内部转账

最少要有：

- 时间
- 金额
- 转出账户
- 转入账户
- 备注

## __AGENT_NAME__ 学习记账习惯的原则

- 优先读取当前设置里的：
  - 资金来源
  - 账户
  - 项目
  - 类别
- 如果你之前已经确认过某个商户、用途或表达习惯，__AGENT_NAME__ 应优先复用上一次确认后的映射
- 如果同一句话还能映射到多个账户、项目或类别，__AGENT_NAME__ 不能猜，必须继续追问
- 只有当金额、类型、账户、项目/类别或资金来源都足够稳定时，才允许真正写入

## 推荐口令模板

### 支出

```text
请记一笔支出：今天午饭 32 元，账户生活账户，分类外卖饮食，时间今天 12:30，备注工作日午餐。
```

### 收入

```text
请记一笔收入：小红书回款 2300 元，入账到经营账户，资金来源小红书，时间现在，备注 3 月回款。
```

### 内部转账

```text
请记一笔内部转账：从经营账户转 2000 元到生活账户，时间现在，备注本月生活补充。
```

## __AGENT_NAME__ 应该如何理解用户输入

### 1. 收入

出现这些表达时，优先判为收入：

- 收到
- 入账
- 到账
- 提现
- 回款
- 红包
- 注资

### 2. 支出

出现这些表达时，优先判为支出：

- 花了
- 买了
- 付了
- 发了
- 交了
- 充值
- 订阅

### 3. 内部转账

出现这些表达时，优先判为转账：

- 从 A 转到 B
- 调拨
- 转入
- 转出
- 补到

内部转账不算收入，也不算支出。

## 当前默认账户

- `经营账户`
- `生活账户`
- `应急账户`

如果用户没有明确说明账户：

- 工作类支出默认 `经营账户`
- 生活类支出默认 `生活账户`
- 红包、家人往来默认 `生活账户`
- 平台回款默认 `经营账户`

## 当前匹配口径

### 资金来源

- `经营`
- `红包`
- `注资`
- `往来`
- 或你在设置里维护的具体来源，如 `小红书`、`B站`、`抖音`

### 项目

- 第 4 层直接来自设置页里的 `项目管理`
- 每个项目只有两项主信息：`项目名称`、`项目信息`
- 例如：`公司运营`、`必要开销`、`额外开销`、`A项目`、`B项目`

### 第 5 层类别

- 第 5 层直接来自设置页里的 `类别管理`
- 每个类别只绑定一个项目
- 所以 __AGENT_NAME__ 读到类别后，必须同时确认它属于哪个项目

如果无法稳定判断：

- 先读取设置里的账户、项目、资金来源、类别
- 优先匹配最接近的已有项
- 仍然不确定时先追问，不要猜

## __AGENT_NAME__ 的输出要求

记账成功后，必须回报这些信息：

- 标题
- 金额
- 类型
- 账户
- 分类
- 时间
- 交易 ID

不要只说“已经记好了”。

## 不应该直接写入的场景

下面这些情况，不要真的记账：

- 用户在讨论分类规则
- 用户在讨论制度或口径
- 用户在讨论 UI 或报表
- 金额缺失
- 账户和类型都高度不明确

这类场景应该先追问，再写入。

## 最小验证动作

当用户说“帮我记一笔”时，`__AGENT_NAME__` 的最小闭环是：

1. 解析类型、金额、账户、分类、时间
2. 调用 `finance-node-bookkeeper`
3. 写入节点
4. 回报结构化结果

当用户说“帮我查一下”时，优先用：

```bash
python3 skills/finance-node-bookkeeper/scripts/finance_node_cli.py config
```

或：


或：

```bash

```
